## 1. Testing Framework Overview

Skill quality verification is a combination of **qualitative evaluation** and **quantitative evaluation**.

| Evaluation Type | Method | Suitable Skills |
|----------|------|-----------|
| **Qualitative** | User directly reviews outputs | Writing style, design, creative works — subjective quality |
| **Quantitative** | Assertion-based automated scoring | File generation, data extraction, code generation — objectively verifiable |

Core loop: **Write → Run test → Evaluate → Improve → Retest**

## 2. Writing Test Prompts

### Principles

Test prompts should be **specific and natural sentences that a real user would actually type**. Abstract or artificial prompts have low test value.

### Bad Examples


"Process the PDF"
"Extract the data"
"Generate a chart"


### Good Examples


"Add a profit margin (%) column to Q4_Sales_Final_v2.xlsx in the Downloads folder
using column C (Revenue) and column D (Cost). Then sort by profit margin
in descending order."



"Extract the table on page 3 from this PDF and convert it to CSV. The table
headers are in 2 rows — the first row is the category, the second row is
the actual column names."


### Prompt Diversity

- Mix **formal / casual** tones
- Mix **explicit / implicit** intent
- Mix **simple / complex** tasks
- Include some with abbreviations, typos, casual expressions

### Coverage

Start with 2–3 prompts, designed to cover:
- 1 core use case
- 1 edge case
- (Optional) 1 compound task

## 3. Execution Testing: With-skill vs Baseline

### 3-1. Comparison Execution Structure

For each test prompt, run two Tasks **simultaneously**:

> **Model configuration:** The `model` parameter is specified in `provider/model` format. Use models configured in the user's OpenCode settings (config). Examples: `anthropic/claude-sonnet-4-20250514`, `openrouter/minimax/minimax-m2.7`, etc.

**With-skill execution:**

Task({ name: "general", prompt: "{test prompt}", model: "{provider/model}", skill: "{skill name}" })
Output path: _workspace/iteration-N/eval-{id}/with_skill/outputs/


**Baseline execution:**

Task({ name: "general", prompt: "{test prompt}", model: "{provider/model}" })
Output path: _workspace/iteration-N/eval-{id}/without_skill/outputs/


### 3-2. Baseline Selection

| Situation | Baseline |
|------|----------|
| New skill creation | Run same prompt without skill |
| Existing skill improvement | Previous skill version (preserve snapshot) |

### 3-3. Timing Data Capture

Save total_tokens and duration_ms from Task results **immediately**. This data is only accessible at the time results are received and cannot be recovered later.

json
{
  "total_tokens": 84852,
  "duration_ms": 23332,
  "total_duration_seconds": 23.3
}

## 4. Quantitative Evaluation: Assertion-based Scoring

### 4-1. Writing Assertions

When outputs are objectively verifiable, define assertions for automated scoring.

**Good assertions:**
- Objectively determinable as true/false
- Descriptive names that make it clear what is being checked just from the results
- Verify the skill's core value

**Bad assertions:**
- Always pass regardless of skill presence (e.g., "output exists")
- Require subjective judgment (e.g., "well written")

### 4-2. Programmatic Verification

If assertions can be verified by code, write them as scripts. Faster and more reliable than manual checking, and reusable across iterations.

### 4-3. Non-discriminating Assertion Warning

Assertions that "pass 100% in both configurations" do not measure the skill's differential value. When such assertions are found, remove them or replace with more challenging ones.

### 4-4. Scoring Result Schema

json
{
  "expectations": [
    {
      "text": "Profit margin column was added",
      "passed": true,
      "evidence": "Confirmed profit_margin_pct column in column E"
    },
    {
      "text": "Sorted by profit margin in descending order",
      "passed": false,
      "evidence": "Original order maintained without sorting"
    }
  ],
  "summary": {
    "passed": 1,
    "failed": 1,
    "total": 2,
    "pass_rate": 0.50
  }
}

## 5. Utilizing Specialized Agents

Quality improves when specialized role agents are used during the testing/evaluation process.

### 5-1. Grader

Performs assertion-based scoring, extracts verifiable claims from outputs, and cross-validates them.

**Role:**
- Pass/fail determination per assertion + evidence provided
- Extract factual claims from outputs and verify them
- Feedback on eval quality itself (suggestions when assertions are too easy or ambiguous)

### 5-2. Comparator (Blind Comparator)

Anonymizes two outputs as A/B and judges quality without knowing which used the skill.

**When to use:** When you want to rigorously confirm "Is the new version really better?" Can be skipped during normal iterative improvement.

**Judgment criteria:**
- Content: accuracy, completeness
- Structure: organization, formatting, usability
- Overall score

### 5-3. Analyzer

Analyzes statistical patterns from benchmark data:
- Non-discriminating assertions (both configurations pass → no differential power)
- High-variance evals (results vary significantly across runs → unstable)
- Time/token tradeoffs (skill improves quality but also increases cost)

## 6. Iterative Improvement Loop

### 6-1. Feedback Collection

Show outputs to the user and collect feedback. Empty feedback is interpreted as "no issues."

### 6-2. Improvement Principles

1. **Generalize feedback** — Narrow fixes that only work for the test example are overfitting. Fix at the principle level.
2. **Remove what doesn't pull its weight** — Read the transcripts; if the skill is making agents do unproductive work, delete that part.
3. **Explain the Why** — Even if user feedback is brief, understand why it matters and reflect that understanding in the skill.
4. **Bundle repetitive work** — If the same helper script is generated in every test run, include it in scripts/ beforehand.

### 6-3. Iteration Procedure


1. Modify the skill
2. Re-run all test cases in a new iteration-N+1/ directory
3. Present results to user (compared with previous iteration)
4. Collect feedback
5. Modify again → repeat


**Termination conditions:**
- User is satisfied
- All feedback is empty (all outputs have no issues)
- No meaningful improvement possible

### 6-4. Draft → Review Pattern

When modifying a skill, write a draft then **re-read it with fresh eyes** and improve. Don't try to write perfectly in one pass; use a draft-review cycle.

## 7. Description Trigger Verification

### 7-1. Trigger Eval Query Writing

Write 20 eval queries — 10 should-trigger + 10 should-NOT-trigger.

**Query quality criteria:**
- Specific and natural sentences that a real user would type
- Include specific details like file paths, personal context, column names, company names
- Mix of length, tone, and format
- Focus on **edge cases** rather than clear-cut answers

**Should-trigger queries (8–10):**
- Same intent expressed in various ways (formal/casual)
- Cases where the skill/file type isn't explicitly mentioned but is clearly needed
- Niche use cases
- Cases that compete with other skills but this skill should win

**Should-NOT-trigger queries (8–10):**
- **Near-misses are key** — Queries with similar keywords but where a different tool/skill is appropriate
- Obviously unrelated queries ("write a Fibonacci function") have no test value
- Adjacent domains, ambiguous expressions, keyword overlap but different context

### 7-2. Existing Skill Conflict Verification

Verify that the new skill's description doesn't overlap with existing skills' trigger areas:

1. Collect descriptions from existing skill list
2. Confirm new skill's should-trigger queries don't incorrectly trigger existing skills
3. If conflicts are found, describe boundary conditions in the description more clearly

## 8. Workspace Structure

Directory structure for systematically managing test/evaluation results:


{skill-name}-workspace/
├── iteration-1/
│   ├── eval-descriptive-name-1/
│   │   ├── eval_metadata.json
│   │   ├── with_skill/
│   │   │   ├── outputs/
│   │   │   ├── timing.json
│   │   │   └── grading.json
│   │   └── without_skill/
│   │       ├── outputs/
│   │       ├── timing.json
│   │       └── grading.json
│   ├── eval-descriptive-name-2/
│   │   └── ...
│   └── benchmark.json
├── iteration-2/
│   └── ...
└── evals/
    └── evals.json


**Rules:**
- Eval directories use **descriptive names** not numbers (e.g., eval-multi-page-table-extraction)
- Each iteration is preserved in an independent directory (never overwrite previous iterations)
- _workspace/ is never deleted — retained for post-verification and audit trails
