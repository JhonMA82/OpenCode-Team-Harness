## 1. Testing framework overview

Skill quality validation combines **qualitative evaluation** and **quantitative evaluation**.

| Evaluation type | Method | Suitable skills |
|----------|------|-----------|
| **Qualitative** | User reviews the artifacts directly | Style, design, creative work, and other subjective quality |
| **Quantitative** | Assertion-based automatic grading | File creation, data extraction, code generation, and other objectively verifiable work |

Core loop: **write → test run → evaluate → improve → retest**

## 2. How to write test prompts

### Principles

Test prompts MUST be **specific, natural sentences that real users might enter**. Abstract or artificial prompts have low test value.

### Bad examples

"Process a PDF"
"Extract data"
"Create a chart"

### Good examples

"Use columns C (revenue) and D (cost) in Q4_revenue_final_v2.xlsx in my Downloads folder to add a profit margin (%) column. Then sort descending by profit margin."

"Extract the table on page 3 of this PDF and convert it to CSV. The table header has two rows: the first row is the category and the second row is the actual column name."

### Prompt diversity

- Mix **formal / casual** tones.
- Mix **explicit / implicit** intent.
- Mix **simple / complex** tasks.
- Include some abbreviations, typos, and casual wording.

### Coverage

Start with 2–3 prompts, but design them to cover:
- 1 core use case
- 1 edge case
- (optional) 1 compound task

## 3. Execution testing: With-skill vs Baseline

### 3-1. Comparison run structure

For each test prompt, run two Tasks **at the same time**:

> **Model setting:** Set the `model` parameter in `provider/model` format. Use a model configured in the user's OpenCode config. Examples: `anthropic/claude-sonnet-4-20250514`, `openrouter/minimax/minimax-m2.7`.

**With-skill run:**

Task({ name: "general", prompt: "{test prompt}", model: "{provider/model}", skill: "{skill name}" })
Output path: _workspace/iteration-N/eval-{id}/with_skill/outputs/


**Baseline run:**

Task({ name: "general", prompt: "{test prompt}", model: "{provider/model}" })
Output path: _workspace/iteration-N/eval-{id}/without_skill/outputs/


### 3-2. Choosing a baseline

| Situation | Baseline |
|------|----------|
| New skill creation | Same prompt without the skill |
| Existing skill improvement | Previous skill version (preserve a snapshot) |

### 3-3. Timing data capture

Save `total_tokens` and `duration_ms` **immediately** from the Task result. This data is only available when the result is received and cannot be recovered later.

json
{
  "total_tokens": 84852,
  "duration_ms": 23332,
  "total_duration_seconds": 23.3
}

## 4. Quantitative evaluation: assertion-based grading

### 4-1. Writing assertions

When artifacts can be objectively validated, define assertions for automatic grading.

**Good assertions:**
- Can be objectively judged true/false.
- Have descriptive names so the result clearly shows what is being checked.
- Validate the skill's core value.

**Bad assertions:**
- Always pass regardless of skill usage (example: "output exists").
- Require subjective judgment (example: "well written").

### 4-2. Programmable validation

If an assertion can be validated with code, write it as a script. This is faster and more reliable than visual inspection, and can be reused across iterations.

### 4-3. Watch for non-discriminating assertions

Assertions that pass 100% in both configurations do not measure the skill's differentiating value. When you find them, remove them or replace them with more challenging assertions.

### 4-4. Grading result schema

json
{
  "expectations": [
    {
      "text": "profit margin column was added",
      "passed": true,
      "evidence": "confirmed profit_margin_pct column in column E"
    },
    {
      "text": "sorted descending by profit margin",
      "passed": false,
      "evidence": "original order was preserved without sorting"
    }
  ],
  "summary": {
    "passed": 1,
    "failed": 1,
    "total": 2,
    "pass_rate": 0.50
  }
}

## 5. Using specialist agents

Using specialist-role agents during testing/evaluation improves quality.

### 5-1. Grader

Performs assertion-based grading and extracts verifiable claims from artifacts for cross-checking.

**Role:**
- Judge pass/fail for each assertion and provide evidence.
- Extract and validate factual claims from artifacts.
- Provide feedback on eval quality itself (for example, assertions that are too easy or ambiguous).

### 5-2. Comparator

Anonymize two artifacts as A/B and judge quality without knowing which one used the skill.

**When to use:** when you need strict confirmation that "the new version is really better." Optional for normal iterative improvement.

**Judgment criteria:**
- Content: accuracy, completeness
- Structure: organization, formatting, usability
- Overall score

### 5-3. Analyzer

Analyzes statistical patterns in benchmark data:
- Non-discriminating assertions (both configurations pass → no discrimination)
- High-variance evals (results vary greatly between runs → unstable)
- Time/token tradeoffs (skill improves quality but also increases cost)

## 6. Iterative improvement loop

### 6-1. Collect feedback

Show artifacts to the user and collect feedback. Treat empty feedback as "no issues."

### 6-2. Improvement principles

1. **Generalize feedback** — narrow fixes fitted only to test examples are overfitting. Fix at the principle level.
2. **Remove weight that does not earn its keep** — read transcripts and delete parts that make agents do unproductive work.
3. **Explain why** — even when user feedback is concise, understand why it matters and reflect that understanding in the skill.
4. **Bundle repeated work** — if every test run creates the same helper script, include it under scripts/ ahead of time.

### 6-3. Iteration procedure

1. Modify the skill.
2. Rerun every test case in a new iteration-N+1/ directory.
3. Present results to the user (compared with the previous iteration).
4. Collect feedback.
5. Modify again → repeat.

**Exit conditions:**
- The user is satisfied.
- All feedback is empty (all artifacts have no issues).
- No further meaningful improvement exists.

### 6-4. Draft → reread pattern

When modifying a skill, write a draft and then **reread it with fresh eyes** to improve it. Do not try to make it perfect in one pass; use a draft-review cycle.

## 7. Description trigger validation

### 7-1. Write trigger eval queries

Write 20 eval queries — 10 should-trigger + 10 should-NOT-trigger.

**Query quality criteria:**
- Specific, natural sentences a real user might enter.
- Include concrete details such as file paths, personal context, column names, and company names.
- Mix length, tone, and format.
- Focus on **boundary cases (edge cases)** rather than obvious answers.

**Should-trigger queries (8–10):**
- Different expressions of the same intent (formal/casual).
- Cases where the skill/file type is not named explicitly but is clearly needed.
- Less common use cases.
- Cases where this skill competes with another skill but should win.

**Should-NOT-trigger queries (8–10):**
- **Near-misses are key** — queries with similar keywords where another tool/skill is more appropriate.
- Clearly unrelated queries ("write a Fibonacci function") have no test value.
- Adjacent domains, ambiguous wording, and keyword overlap with different context.

### 7-2. Validate conflicts with existing skills

Confirm the new skill's description does not overlap with existing skill trigger areas:

1. Collect descriptions from the existing skill list.
2. Confirm the new skill's should-trigger queries do not incorrectly trigger existing skills.
3. If conflicts are found, make the description's boundary conditions more explicit.

## 8. Workspace structure

Directory structure for managing test/evaluation results systematically:

{skill-name}-workspace/
├── iteration-1/
│   ├── eval-descriptive-name-1/
│   │   ├── eval_metadata.json
│   │   ├── with_skill/
│   │   │   ├── outputs/
│   │   │   ├── timing.json
│   │   │   └── grading.json
│   │   └── without_skill/
│   │       ├── outputs/
│   │       ├── timing.json
│   │       └── grading.json
│   ├── eval-descriptive-name-2/
│   │   └── ...
│   └── benchmark.json
├── iteration-2/
│   └── ...
└── evals/
    └── evals.json


**Rules:**
- Use **descriptive names** for eval directories, not numbers (example: eval-multi-page-table-extraction).
- Preserve each iteration in an independent directory (do not overwrite previous iterations).
- Do not delete _workspace/ — it is used for post-hoc verification and audit trail.
