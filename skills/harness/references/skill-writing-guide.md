## 1. Description Writing Patterns

The description is the skill's sole trigger mechanism. Claude sees only name + description from the available_skills list and decides whether to use the skill.

### Understanding the Trigger Mechanism

Claude tends not to invoke skills for simple tasks it can easily handle with its built-in tools. Simple requests like "read this PDF" may not trigger even with a perfect description. The more complex, multi-step, and specialized the task, the higher the skill trigger probability.

### Writing Principles

1. Describe both **what the skill does** + **specific trigger situations**
2. Specify boundary conditions that distinguish cases that are similar but should NOT trigger
3. Be slightly "pushy" — compensate for Claude's tendency to be conservative about triggering

### Good Examples

yaml
description: "Performs all PDF operations including reading, text/table extraction,
  merging, splitting, rotating, watermarking, encryption/decryption, and OCR.
  Always use this skill when a .pdf file is mentioned or PDF output is requested.
  Especially useful when transformation/editing/analysis is needed, not just
  a simple 'read this PDF' request."


yaml
description: "All spreadsheet operations for Excel/CSV/TSV files including
  column addition, formula calculation, formatting, charting, and data cleaning.
  Use this skill whenever the user mentions a spreadsheet file — even casually
  ('the xlsx in the Downloads folder')."


### Bad Examples

- "A skill that processes data" — Too vague, unclear what files/operations
- "PDF-related operations" — No specific actions listed, trigger situations not described

## 2. Body Writing Style

### Why-First Principle

LLMs make correct judgments on edge cases when they understand the reasoning. Context delivery is more effective than coercive rules.

**Bad example:**
markdown
ALWAYS use pdfplumber for table extraction. NEVER use PyPDF2 for tables.


**Good example:**
markdown
Use pdfplumber for table extraction. PyPDF2 specializes in text extraction
and does not preserve table row/column structure. pdfplumber recognizes cell
boundaries and returns structured data.


### Generalization Principle

When problems are found in feedback or test results, **generalize at the principle level** instead of making narrow fixes that only work for a specific example.

**Overfitting fix:**
markdown
If there's a "Q4 Revenue" column, convert that column to numeric.


**Generalized fix:**
markdown
If a column name contains keywords implying numeric values such as "revenue",
"amount", or "quantity", convert that column to numeric type. Preserve the
original value on conversion failure.


### Imperative Tone

Use imperative forms like "Do X", "Use Y" instead of "it can be done" or "one may use". A skill is an instruction document.

### Context Conservation

The context window is a shared resource. Ask whether every sentence justifies its token cost:
- "Does Claude already know this?" → Delete
- "Would Claude make a mistake without this explanation?" → Keep
- "Would one concrete example be more effective than a long explanation?" → Replace with example

## 3. Output Format Definition Patterns

Use when the format of outputs is important for a skill:

markdown
## Report Structure
Follow this template exactly:

# [Title]
## Summary
## Key Findings
## Recommendations


Keep format definitions concise; including actual examples makes them more effective.

## 4. Example Writing Patterns

Examples are more effective than long explanations:

markdown
## Commit Message Format

**Example 1:**
Input: Add JWT token-based user authentication
Output: feat(auth): implement JWT-based authentication

**Example 2:**
Input: Fix bug where password show button doesn't work on login page
Output: fix(login): fix password visibility toggle button behavior

## 5. Progressive Disclosure Patterns

### Pattern 1: Domain-based Separation


bigquery-skill/
├── SKILL.md (overview + domain selection guide)
└── references/
    ├── finance.md (revenue, billing metrics)
    ├── sales.md (opportunities, pipeline)
    └── product.md (API usage, features)


When the user asks about revenue, load only finance.md.

### Pattern 2: Conditional Detail

markdown
# DOCX Processing

## Document Creation
Create new documents with docx-js. → See [DOCX-JS.md](references/docx-js.md).

## Document Editing
For simple edits, modify XML directly.
**If track changes are needed**: See [REDLINING.md](references/redlining.md)


### Pattern 3: Large Reference File Structure

Reference files over 300 lines should include a table of contents at the top:

markdown
# API Reference

## Table of Contents
1. [Authentication](#authentication)
2. [Endpoint List](#endpoint-list)
3. [Error Codes](#error-codes)
4. [Rate Limits](#rate-limits)

## 6. Script Bundling Decision Criteria

Observe agent transcripts during test runs. Bundle when these patterns appear:

| Signal | Action |
|------|------|
| Same helper script generated in 3 out of 3 tests | Bundle in scripts/ |
| Same pip install/npm install executed every time | Specify dependency installation step in skill |
| Same multi-step approach repeated | Describe as standard procedure in skill body |
| Same error followed by same workaround every time | Describe known issue and solution in skill |

Bundled scripts must pass execution testing.

## 7. Data Schema Standards

Standard schemas for consistency in data exchange between skills. Can be used for testing/evaluation of skills generated by the harness.

### eval_metadata.json

Metadata for each test case:

json
{
  "eval_id": 0,
  "eval_name": "descriptive-name-here",
  "prompt": "User's task prompt",
  "assertions": [
    "Output contains X",
    "File was generated in Y format"
  ]
}


### grading.json

Assertion-based scoring results:

json
{
  "expectations": [
    {
      "text": "Output contains 'Seoul'",
      "passed": true,
      "evidence": "Confirmed 'Seoul region data extraction' in step 3"
    }
  ],
  "summary": {
    "passed": 2,
    "failed": 1,
    "total": 3,
    "pass_rate": 0.67
  }
}


**Field name note:** Use exactly text, passed, evidence (no variants like name/met/details).

### timing.json

Execution time/token measurement:

json
{
  "total_tokens": 84852,
  "duration_ms": 23332,
  "total_duration_seconds": 23.3
}


Save total_tokens and duration_ms from Task results immediately. This data is only accessible at the time results are received and cannot be recovered later.

## 8. What NOT to Include in a Skill

- README.md, CHANGELOG.md, INSTALLATION_GUIDE.md, and other supplementary documents
- Meta-information about the skill creation process (test results, iteration history)
- User-facing documentation (skills are instruction documents for AI agents)
- General knowledge that Claude already knows
