## 1. Description writing patterns

The description is the skill's only trigger mechanism. Claude decides whether to use a skill from the `available_skills` list using only name + description.

### Understanding the trigger mechanism

Claude tends not to call skills for simple tasks it can easily handle with its default tools. Simple requests like "read this PDF" may not trigger the skill even with a perfect description. Complex, multi-step, specialized tasks are more likely to trigger a skill.

### Writing principles

1. Describe both **what the skill does** and **the concrete trigger situations**.
2. State boundary conditions that distinguish similar cases that should not trigger the skill.
3. Be somewhat "pushy" to compensate for Claude's conservative trigger decisions.

### Good examples

yaml
description: "Perform all PDF tasks, including reading PDF files, text/table extraction,
  merging, splitting, rotating, watermarking, encryption/decryption, and OCR.
  Use this skill whenever a .pdf file is mentioned or a PDF output is requested.
  Especially useful when conversion, editing, or analysis is needed, not just
  a request to 'read' a PDF."


yaml
description: "All spreadsheet work for Excel/CSV/TSV files, including adding columns,
  calculating formulas, formatting, charts, and data cleaning. If the user mentions
  a spreadsheet file — even casually, like 'the xlsx in Downloads' — use this skill."


### Bad examples

- "A skill for processing data" — too vague; unclear which files/tasks it handles.
- "PDF-related work" — no concrete action list and no trigger situations.

## 2. Body writing style

### Why-first principle

LLMs make better edge-case decisions when they understand the reason. Context is more effective than coercive rules.

**Bad example:**
markdown
ALWAYS use pdfplumber for table extraction. NEVER use PyPDF2 for tables.


**Good example:**
markdown
Use pdfplumber for table extraction because PyPDF2 specializes in text extraction
and cannot preserve table row/column structure. pdfplumber recognizes cell boundaries
and returns structured data.


### Generalization principle

When feedback or test results reveal an issue, **generalize at the principle level** instead of making narrow fixes fitted to one example.

**Overfit fix:**
markdown
If there is a "Q4 revenue" column, convert that column to numbers.


**Generalized fix:**
markdown
If a column name contains keywords that imply numeric values, such as "revenue",
"amount", or "quantity", convert that column to a numeric type. If conversion fails,
preserve the original value.


### Imperative tone

Use direct imperative wording. Skills are instruction documents.

### Context economy

The context window is a shared resource. Ask whether every sentence justifies its token cost:
- "Does Claude already know this?" → delete it.
- "Would Claude make a mistake without this explanation?" → keep it.
- "Would one concrete example work better than a long explanation?" → replace with an example.

## 3. Output format definition patterns

Use this when the artifact's format matters:

markdown
## Report structure
Follow this template exactly:

# [Title]
## Summary
## Key findings
## Recommendations


Keep format definitions concise. Including a real example is often more effective.

## 4. Example writing patterns

Examples are more effective than long explanations:

markdown
## Commit message format

**Example 1:**
Input: add JWT token-based user authentication
Output: feat(auth): implement JWT-based authentication

**Example 2:**
Input: fix a bug where the password visibility button does not work on the login page
Output: fix(login): restore password visibility toggle behavior

## 5. Progressive Disclosure patterns

### Pattern 1: Domain-specific split


bigquery-skill/
├── SKILL.md (overview + domain selection guide)
└── references/
    ├── finance.md (revenue, billing metrics)
    ├── sales.md (opportunities, pipeline)
    └── product.md (API usage, features)


If the user asks about revenue, load only finance.md.

### Pattern 2: Conditional detail

markdown
# DOCX handling

## Document creation
Create new documents with docx-js. → see [DOCX-JS.md](references/docx-js.md).

## Document editing
For simple edits, modify XML directly.
**If tracked changes are needed**: see [REDLINING.md](references/redlining.md).


### Pattern 3: Large reference file structure

Reference files over 300 lines include a table of contents at the top:

markdown
# API Reference

## Table of Contents
1. [Authentication](#authentication)
2. [Endpoint list](#endpoint-list)
3. [Error codes](#error-codes)
4. [Rate limits](#rate-limits)

## 6. Script bundling criteria

Observe agent transcripts during test runs. Bundle scripts when these patterns appear:

| Signal | Action |
|------|------|
| The same helper script is created in 3 out of 3 tests | Bundle it in scripts/ |
| The same `pip install`/`npm install` runs every time | Document dependency installation in the skill |
| The same multi-step approach repeats | Describe it as the standard procedure in the skill body |
| The same workaround follows similar errors every time | Document known issues and solutions in the skill |

Bundled scripts MUST be execution-tested.

## 7. Data schema standards

Use standard schemas to keep data exchange between skills consistent. These can be used for testing/evaluation of skills generated by the harness.

### eval_metadata.json

Metadata for each test case:

json
{
  "eval_id": 0,
  "eval_name": "descriptive-name-here",
  "prompt": "user's task prompt",
  "assertions": [
    "the artifact contains X",
    "a file was created in Y format"
  ]
}


### grading.json

Assertion-based grading result:

json
{
  "expectations": [
    {
      "text": "the artifact contains 'Seoul'",
      "passed": true,
      "evidence": "confirmed 'Seoul region data extraction' in step 3"
    }
  ],
  "summary": {
    "passed": 2,
    "failed": 1,
    "total": 3,
    "pass_rate": 0.67
  }
}


**Field name caution:** use exactly `text`, `passed`, and `evidence` (do not use variants such as `name`, `met`, or `details`).

### timing.json

Runtime/token measurement:

json
{
  "total_tokens": 84852,
  "duration_ms": 23332,
  "total_duration_seconds": 23.3
}


Save `total_tokens` and `duration_ms` immediately from the Task result. This data is only available when the result is received and cannot be recovered later.

## 8. What not to include in a skill

- Supplemental docs such as README.md, CHANGELOG.md, or INSTALLATION_GUIDE.md.
- Meta information about the skill creation process (test results, iteration history).
- User-facing manuals (skills are instruction documents for AI agents).
- General knowledge Claude already has.
