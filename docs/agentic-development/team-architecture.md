# Agent Team Architecture

> **Purpose**: Document the architecture rationale for agent teams generated by the OpenCode Team Harness.

## Overview

The harness generates **orchestrator-based agent teams** — one orchestrator agent delegates work to specialist agents based on trigger conditions. This is not a mesh or peer-to-peer model; all coordination flows through the orchestrator.

## Architectural Pattern

```
User Request
     │
     ▼
┌──────────────────┐
│   Orchestrator   │  ← Routes work by trigger
│   (AGENTS.md)    │
└──────┬───────────┘
       │
       ├── Agent A (Domain Analysis) ──→ Skill A
       ├── Agent B (Implementation)  ──→ Skill B
       ├── Agent C (Review)          ──→ Skill C
       └── Agent D (Validation)      ──→ Skill D
```

### Why Orchestrator-Based?

| Pattern | Tradeoff | Verdict |
|---------|----------|---------|
| Orchestrator + specialists | Clear roles, single entry point, manageable | **Selected** |
| Mesh / peer-to-peer | Complex coordination, no central authority | Rejected |
| Single monolithic agent | No specialization, bloated prompts | Rejected |

The orchestrator-based pattern aligns with OpenCode's existing architecture (the `task` tool, agent dispatch) and keeps generated teams composable — you can add or remove specialist agents without rewiring the whole team.

## Key Design Decisions

### 1. Deterministic Generation (AG-8, CC-3)

All agent definitions and skills are generated from templates with `{{PLACEHOLDER}}` substitution. No AI model is invoked during generation. This keeps output auditable in Git and reproducible across runs.

### 2. `.opencode/` as Primary Output Path (IR-7, IR-8, SG-4)

Generated files go to `.opencode/agents/` and `.opencode/skills/` by default. If portable compat is enabled, a deterministic mirror writes to `.agents/skills/`. The `.opencode/` path is always authoritative — `.agents/` is a derived artifact.

### 3. No Self-Generation

This harness repo does NOT generate its own agent team. It is the meta-skill — it generates teams for consumer projects. See [design rationale](../specs/opencode-team-harness-mvp/design.md#decision-harness-repo-vs-generated-output).

## Agent Interaction Flow

1. **User invokes** the harness from a consumer project via `skills/harness/SKILL.md`
2. **Domain Analyzer** reads project context (name, description, tech stack, conventions)
3. **Agent Writer** fills `AGENTS.md` and `.opencode/agents/*.md` from templates
4. **Skill Writer** fills `.opencode/skills/*/SKILL.md` from templates
5. **Validator** checks generated files against spec requirements

All agents are stateless — they read context and write deterministic output. There is no persistent agent memory.

## Extending the Team

To add a new agent role:

1. Add a new template at `skills/harness/templates/`
2. Add the agent row to `AGENTS.md` template
3. Add associated skill template
4. Update the validation assertions in `evals/promptfoo/`

This keeps the team extensible without modifying any agent logic — the templates ARE the logic.
